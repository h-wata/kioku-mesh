# Topology PoC: hub-and-spoke transit verification (2026-05-10)

## Goal

「新 peer 追加で全 PC の zenohd 再起動が必要」という運用負担を、Zenoh の topology
で回避できるかを実機 3-PC で検証する。具体的には次の 2 仮説を立証する。

1. **Transit reachability**: spoke が hub に対して `connect` 1 本だけ書けば、他
   spoke のデータも transit 経由で届くか。
2. **Add peer without restart**: 新 peer 追加時に hub / 他 spoke の config
   修正・restart は不要か。

## Setup

3-PC 構成。spoke (WSL2) を新たに追加し、既存の home / office mesh を一切 touch
せず動かす。

| Role | Host | mesh-mem | zenohd config (runtime, untracked) | listen | connect |
|------|------|----------|-----------------------------------|--------|---------|
| hub  | hwatahome-jazzy (home) | 0.2.3 | `/tmp/zenohd_home_runtime.json5` | 127.0.0.1:7447, 192.168.134.28:7447, 192.168.3.12:7447 | tcp/192.168.128.12:7447 (office) |
| spoke (existing) | hwata-jazzy (office) | 0.2.3 | `config/zenohd_office.json5` (WIP, 実 IP) | 127.0.0.1:7447, 192.168.128.12:7447 | tcp/192.168.134.28:7447 (home) |
| spoke (new) | DESKTOP-SRFBMPL (WSL2) | 0.2.3 | `~/.config/mesh-mem/zenohd_wsl_local.json5` | 192.168.3.29:7448 | tcp/192.168.3.12:7447 (hub only) |

WSL2 は loopback / 7447 が Windows host との mirrored networking で衝突したため
listen を `192.168.3.29:7448` のみに変更。clock は `+1.0s bias` の manual step で
home との skew を **-14 ms** に追い込んでから replication を計測 (詳細は
add-mesh-peer skill §9)。

## Method

各 PC から `mesh-mem save --project mesh-topo-exp` で 1 件ずつ observation を作り、
他の 2 PC から `mesh-mem get-memory <id>` で取得できるかを cross-check した。

| Saver | ID |
|-------|----|
| home  | `0e0ace1e03964c7ba9e5253fde9d5245` |
| WSL2  | `fbbbb957771746098deb1ce96b5a0940` |
| office| `9cdf7d452c264251b01cdc0718b15a65` |

## Result: Reachability matrix

| from \ to (saver) | HOME id | WSL2 id | OFFICE id |
|---|---|---|---|
| HOME (getter)   | self ✅ | ❌ timeout (mesh-mem 課題) | ❌ timeout (mesh-mem 課題) |
| WSL2 (getter)   | ❌ timeout (mesh-mem 課題) | self ✅ | **✅ via transit** |
| OFFICE (getter) | ✅ direct | **✅ via transit** | self ✅ |

太字が **直接 link が無い 2 hop transit ペア**。

## Findings

### 1. Transit reachability — 確認 ✅

WSL2 は `connect: [tcp/192.168.3.12:7447]` (hub) のみ。office との直接 link 無し。
それでも:
- WSL2 saved → office から取得 ✅
- office saved → WSL2 から取得 ✅

→ Zenoh router transit は片方向 connect だけで双方向セッションを成立させ、GET
クエリも transit 経由で他 spoke に到達する。**新 spoke は hub 1 本 connect で
全 mesh のデータに届く** という仮説が事実観測された。

### 2. Add peer without restart — 確認 ✅

WSL2 を追加するに当たり:
- home の zenohd: **触らず**
- office の zenohd: **触らず**
- home / office の config: **編集なし**

それでも上記 reachability が成立。仮説どおり、新 spoke 追加で既存 peer の
restart は要らない。

### 3. 例外条件: hub の listen 集約は事前に必要

今回 WSL2 (192.168.3.29) から home にアクセスするのに、home が **既に**
`192.168.3.12:7447` を listen していたため到達できた。もし home が ppp0 IP
(192.168.134.28) しか listen していなかったら、home の listen を 1 度だけ拡張する
必要があった (= hub restart 1 回)。

→ **運用規約**: hub は「将来追加するであろう全 spoke から到達可能な IP」を
あらかじめ listen に集約しておく (LAN / Tailscale / VPN それぞれ)。これは
1 度きりの作業で、新 spoke が増えるたびに繰り返す作業ではない。

## Side findings (topology とは別の課題)

### A. WSL2 mirrored networking の port 衝突

`localhost` / LAN IP は Windows host と stack を共有するため、Windows host 側に
未知のプロセスが 7447 を握っていると WSL2 内 zenohd が `Address already in use
(os error 98)` で bind 失敗する。**`ss -tlnp` / `lsof` / `/proc/net/tcp` のいずれ
にも表示されない** ため原因特定が難しい。回避は port を 7448 にずらす + loopback
listener を消す。

### B. WSL2 clock drift

Windows host hibernation 後に WSL2 clock が秒オーダーでずれる。`systemd-
timesyncd` は active でも step (即時補正) しないため、Zenoh の 500ms timestamp
tolerance を超えて replication が止まる。chrony / ntpdate もデフォルト未
インストール。**`sudo timedatectl set-ntp false; sudo date -u -s @<epoch>` の
manual step が必要**。詳細は add-mesh-peer skill §9-b。

### C. mesh-mem `get-memory` の wildcard scan

home (36 MB store) からの `mesh-mem get-memory <id>` が `mem/obs/**` で query
を流し、各 peer の storage_manager から全 obs を pull してから client side で
ID マッチを探す実装になっている (`src/mesh_mem/store.py:543` 付近の
`_find_by_id_via_zenoh`)。home 自身の reply が大きいため query timeout
(default 10s) に間に合わず failure。

→ topology 起因ではなく mesh-mem 側の query 設計の課題。reachability matrix の
「HOME 行 / 列」が空欄になる原因。**別 issue として切り出して個別追跡推奨**。

## 結論と運用規約への反映

- topology は「1 hub + N spokes」を正規パターンとし、新 spoke 追加で既存 peer
  の config / restart を行わない運用に倒す。
- `config/zenohd_peer.json5.template` を hub-spoke パターンに書き換え済
  (commit でこのレポートと同時に反映)。
- `add-mesh-peer` skill (`~/.claude/skills/add-mesh-peer/SKILL.md`) を更新済:
  - 入力名 `EXISTING_PEER_IP` → `HUB_IP`
  - ステップ 5 を「hub の listen 拡張は 1 度きりの作業」に reframe
  - ステップ 9 (新規) で WSL2 ホスト固有の port 衝突 + clock step 手順を追加
  - トラブルシュート表に WSL2 / clock / get-memory wildcard の症状を追加
- gossip discovery の検証 (Exp 3) は当面不要。N が大きくなった段階で再検討する。
